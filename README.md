# Chessko

체스 오프닝을 공부하고, 연습하고, 모르는 포지션은 엔진과 LLM으로 해설받는 모바일 앱입니다.

기존 체스 앱이 "이 수가 최선"이라고만 알려주고 끝나는 반면, Chessko는 **왜 그 수가 최선인지**를 자연어로 설명합니다. Stockfish의 수치 평가에 오프닝 원칙 텍스트(Vector DB)를 결합한 RAG 해설이 핵심이고, 그 위에 오프닝 학습·연습 기능을 얹은 구조입니다.

## 저장소 구성

세 개의 독립적인 구성요소로 나뉘며, 각 폴더에 자체 README가 있습니다.

| 폴더 | 역할 | 기술 스택 | 문서 |
| --- | --- | --- | --- |
| [`mobile/`](mobile/) | 사용자가 실제로 쓰는 앱. 오프닝 학습, 연습(좌표/퍼즐/퀴즈), 마이페이지, 포지션 분석 | Expo SDK 57, React Native, TypeScript, chess.js | [mobile/README.md](mobile/README.md) |
| [`backend/`](backend/) | 분석 API 서버. Stockfish 프로세스 풀 관리, RAG 해설 생성, Vision 서비스 프록시 | Java 21, Spring Boot 3.3 | [backend/README.md](backend/README.md) |
| [`vision/`](vision/) | 체스 앱 스크린샷 → FEN 변환 마이크로서비스 | Python 3.10+, FastAPI, OpenCV | [vision/README.md](vision/README.md) |

호출 방향은 한 방향입니다:

```text
mobile  ──HTTP──>  backend  ──HTTP──>  vision
                      │
                      ├── Stockfish (로컬 프로세스, UCI)
                      ├── OpenAI API (임베딩 + 챗)
                      └── Supabase pgvector (오프닝 원칙 검색)
```

모바일 앱은 `vision`을 직접 호출하지 않고 항상 `backend`를 거칩니다. 앱에 API 키를 두지 않기 위해서입니다.

## 앱 개념

Lotus Chess 스타일의 하단 3탭 구조입니다. 학습 데이터(저장한 오프닝, 퍼즐 진행도, 연습 기록)는 전부 **기기 로컬(AsyncStorage)**에 저장되며, 별도의 회원가입/로그인은 없습니다.

* **오프닝 탭** — 백/흑 오프닝을 가로 스크롤 캐러셀에서 고르고, 보드 위 화살표를 따라 수순을 한 수씩 익힙니다. 오프닝 하나는 여러 라인(메인라인 / 변형 / **상대의 대표적인 실수를 응징하는 수순**)으로 나뉘고, 수마다 배지(`!!` 탁월수 · `?` 실수 · `??` 블런더 …)와 한국어 해설, 스톡피쉬 평가값이 붙습니다. 상대의 수는 자동으로 둬지고, 수순을 벗어난 수를 두면 화면 테두리가 빨갛게 번쩍이며 자동으로 되돌아갑니다. 라인은 트리로 골라 들어가고, 공부 중에 다른 라인의 수를 두면 그 라인으로 자동으로 넘어갑니다. **보드를 뒤집으면 같은 라인을 반대 색으로** 공부합니다 — 백으로 익힌 수순이 흑 차례에서 그대로 연습 대상이 됩니다. 아무 포지션에서나 "이 포지션 분석하기"로 엔진 분석 화면에 진입할 수 있습니다.
* **연습 탭** — 좌표 연습(리체스 스타일), 메이트 퍼즐 30개, 저장한 오프닝의 다음 수를 맞히는 리콜 퀴즈.
* **마이페이지 탭** — chess.com 닉네임 연동(공개 API, 인증 불필요)으로 최근 대국을 확인하고, 대국을 누르면 수순을 되짚으며 스톡피쉬 이밸류에이션 바로 우위 변화를 봅니다. 로컬 학습 통계도 여기 모입니다. 설정(애니메이션 on/off)은 헤더 우측 톱니바퀴로 들어갑니다.

## 로컬 실행

백엔드 두 서비스(`backend` + `vision`)는 **Docker로 한 번에** 띄우는 것을 권장하고, 모바일은 Expo로 따로 실행합니다. 컨테이너를 쓰지 않으려면 아래 "Docker 없이 직접 실행"을 따르세요.

### 사전 준비

* **Docker + Docker Compose** — 권장 실행 경로. 이 경우 Java·Stockfish·Python을 따로 설치할 필요가 없습니다.
* **Node.js + npm** (모바일, Expo SDK 57) — 컨테이너화 대상이 아니므로 항상 필요합니다.
* (직접 실행 시에만) **Java 21** · **Stockfish 실행 파일** · **Python 3.10+**.
* (선택) **OpenAI API 키**, **Supabase 프로젝트** — RAG 해설용. 없어도 나머지 기능은 정상 동작합니다.

### 1) 백엔드 + Vision — Docker (권장)

```bash
cp .env.example .env           # 최초 1회 — 개발용으로 BACKEND_BIND=0.0.0.0 주석 해제
docker compose up --build      # 백엔드 API가 :8080, 내부 네트워크에서 vision:8000 호출
```

**포트는 기본값이 `127.0.0.1`(로컬 전용)입니다.** 공개 호스트에서 Caddy 뒤에 두기 위한 기본값이라, 그대로 두면 **같은 와이파이의 실기기에서 접속이 거부됩니다.** 폰으로 테스트하려면 루트 `.env`에 `BACKEND_BIND=0.0.0.0`을 넣으세요. 브라우저(`npm run web`)만 쓴다면 기본값으로 충분합니다.

`backend/.env`의 값(비밀키 등)을 그대로 읽습니다. `OPENAI_API_KEY`가 비어 있으면 RAG 해설만 503을 반환하고 **스톡피쉬 분석·대국 복기·스크린샷 스캔은 정상 동작합니다.** Vision의 등록 테마는 볼륨에 영속됩니다. 실행 상세와 HTTPS 배포(Caddy)는 [deploy/README.md](deploy/README.md)를 참고하세요.

### 1-B) 백엔드 + Vision — Docker 없이 직접 실행

`vision` → `backend` 순서로, 각자 다른 터미널에서 실행합니다.

```bash
# Vision (vision/) — 첫 사용 전 테마 등록 필요, vision/README.md 참고
cd vision && pip install -r requirements.txt && uvicorn app.main:app --reload --port 8000

# 백엔드 (backend/) — .env에 최소 STOCKFISH_PATH 필요, backend/README.md 참고
cd backend && ./run.sh          # .env를 읽어 export한 뒤 ./gradlew bootRun
```

### 2) 모바일 (`mobile/`)

```bash
cd mobile
npm install
npm run web       # 기기/에뮬레이터는 `npm start` 후 Expo Go로 QR 스캔
```

`mobile/.env`의 `EXPO_PUBLIC_API_BASE_URL`이 백엔드를 가리켜야 합니다. 실기기에서는 `localhost`가 기기 자신을 의미하므로 PC의 LAN IP를 넣으세요 — [mobile/README.md](mobile/README.md) 참고.

### 테스트

```bash
cd backend && ./gradlew test          # JUnit (Java 21 필요)
cd vision  && python -m pytest        # requirements-dev.txt 설치 후
cd mobile  && npm run typecheck       # 앱 + scripts 타입 체크
```

## 주요 고려사항 및 리스크

* **Stockfish 프로세스 관리** — 요청이 몰리면 엔진 프로세스가 무한히 생성돼 서버가 죽을 수 있습니다. `StockfishService`가 고정 크기 풀로 인스턴스를 관리하고, 대여 타임아웃을 둬 통제합니다.
* **프롬프트 인젝션** — FEN 입력란에 임의 텍스트를 넣어 LLM을 조작하지 못하도록 `BoardState`가 FEN을 엄격히 검증한 뒤에만 엔진·LLM에 전달합니다.
* **예외 처리** — 이미지 인식 실패, 엔진 타임아웃, 외부 LLM 장애 등은 각각 별도 예외로 구분해 사용자에게 명확한 한국어 메시지로 내려줍니다(`GlobalExceptionHandler`).

## 개발 진행 현황

* **Phase 1 완료 — 엔진 코어 (`backend/`)** — Spring Boot + Stockfish 프로세스 풀 기반 `/api/v1/analysis`. FEN 검증, 체크메이트/스테일메이트 감지, 엔진 타임아웃 처리.
* **Phase 2 완료 — 모바일 앱 (`mobile/`)** — Expo 기반 FEN 입력, 체스판 렌더링, 기보 뷰어, 백엔드 분석 연동.
* **Phase 3 완료 — RAG 해설 (`backend/`)** — `/api/v1/analysis/commentary`. Stockfish 평가 + Supabase pgvector 검색 + OpenAI 챗으로 한국어 해설 생성. 체크메이트/스테일메이트는 LLM 호출 없이 고정 문구로 응답.
* **Phase 4 완료 — Vision 스캐너 (`vision/`, `backend/`, `mobile/`)** — 실제 체스판 사진이 아니라 **앱 스크린샷**을 입력으로 받도록 스코프를 조정했습니다. 스크린샷은 원근 왜곡·조명 변화가 없고 기물 스타일이 고정돼 있어, 학습 모델이나 클라우드 API 없이 순수 OpenCV만으로 충분히 정확합니다.
* **Phase 5 완료 — Lotus 스타일 앱 개편 (`mobile/`, `backend/`)** — 단일 FEN 분석기 화면을 하단 3탭 구조(오프닝 / 연습 / 마이페이지)로 재구성했습니다.
  * 오프닝: 백/흑 캐러셀, 화살표 가이드(상대 수는 자동 재생), 북마크. 오프닝 20개는 chess.js로 합법성을 검증했습니다.
* **Phase 6 완료 — 오프닝 다중 라인 + 수별 평가 (`mobile/`)** — 오프닝 하나가 라인 여러 개(메인/변형/실수 응징)를 갖도록 데이터 모델을 바꾸고, 수마다 등급 배지·해설·평가값을 붙였습니다. 평가는 `npm run gen:openings`가 로컬 스톡피쉬(depth 18)로 미리 구워 커밋하므로 앱 런타임에는 엔진 호출이 없습니다. `!!`/`!`는 엔진이 만들지 않고 데이터 작성자가 SAN 접미사로 지정합니다.
  * 연습: 좌표 연습, 메이트 퍼즐 30개(정답 유일성을 chess.js 전수 탐색으로 보장), 저장한 오프닝 리콜 퀴즈.
  * 마이페이지: chess.com 공개 API 연동, 최근 대국 목록, 대국 복기(PGN 재생 + 이밸류에이션 바), 학습 기록, 설정.
  * 백엔드: 대국 전체를 한 번에 평가하는 `/api/v1/analysis/game` 추가(엔진 풀에 병렬 팬아웃, 포지션당 200ms).
  * 화면 진입 안착과 기물 이동 애니메이션은 설정에서 끌 수 있습니다.
* **Phase 7 완료 — 라인 트리 + 저작 도구 (`mobile/`)** — 라인을 평평한 목록이 아니라 **트리**로 다룹니다. 트리는 별도 데이터가 아니라 수순의 공통 접두사와 `branchPly`에서 계산되므로(`openingTree.ts`), 라인을 추가하면 저절로 제자리에 붙습니다. 공부 중 다른 라인의 수를 두면 자동 전환되고, 갈림목에서 끝나는 라인은 이어지는 갈래를 골라 계속 갑니다. **보드를 뒤집으면 같은 라인을 반대 색으로** 공부합니다 — 라인마다 데이터가 필요 없어 전 오프닝에 적용됩니다. 데이터는 `npm run author` 편집기로 만들며(오프닝·갈래 추가·수정·삭제, 수별 등급, 설명·레퍼런스), 큐레이션 원본 `openings.ts`는 도구가 고치지 않고 옆의 JSON에 덧씌우는 구조입니다([mobile/scripts/README.md](mobile/scripts/README.md)).
* **배포 준비 완료 — 컨테이너화 + 공개 노출 대비** — `backend`(런타임 이미지에 Stockfish 포함)와 `vision`을 `docker compose`로 묶고, HTTPS는 `tls` 프로파일의 Caddy가 처리합니다([deploy/README.md](deploy/README.md)). 공개 배포를 앞두고 세 가지를 손봤습니다: 계정 없는 API라 **주소별 요청 제한**(비싼 엔드포인트에 더 큰 비용), `/health`와 컨테이너 헬스체크, 그리고 8080을 loopback으로 내려 Caddy를 우회할 수 없게 한 것. **앱은 EAS 빌드 설정을 마쳤습니다**([mobile/README.md](mobile/README.md#출시-빌드-eas)) — 남은 건 실제 호스트 확보와 스토어 등록으로, 아래 "출시 준비" 참고.
* **다음 단계** — 앱 다듬기(콘텐츠 채우기), 아래 "출시 준비"와 "커뮤니티 라인" 계획, 그리고 Freemium/결제(인앱 구독, 호출 횟수 제어).

## 출시 준비

### 순서를 이렇게 잡은 이유

작업량이 아니라 **대기 시간**이 일정을 정합니다.

* **구글 플레이 비공개 테스트가 달력을 잡아먹습니다.** 개인 개발자 계정은 프로덕션 출시 전에 테스터 12명으로 비공개 테스트를 14일 연속 돌려야 합니다(정책은 콘솔에서 현재 조건 확인). 코드 품질과 무관한 순수 대기라, **먼저 돌려놓고 그동안 다듬는 것이 순수 이득**입니다.
* **백엔드 없이도 테스트를 시작할 수 있습니다.** 오프닝 학습·퍼즐·연습은 전부 오프라인이고, 분석·복기·스캔만 백엔드를 씁니다. 호스트 확보가 늦어져도 테스트 일정이 밀리지 않습니다.
* **커뮤니티는 앱의 같은 화면을 바꿉니다.** 지금 수마다 설명 하나를 보여주는 자리가 나중에 목록 + 좋아요 + 게시판이 됩니다. 그 부분을 지금 정성껏 다듬으면 다시 만들게 됩니다.

## 커뮤니티 라인 계획

큐레이션만으로는 라인을 다 채울 수 없습니다. 그래서 기여 단위를 **"수 하나에 붙는 설명"** 으로 잡습니다 — 수순은 사실이라 금방 중복되지만, 그 수를 어떻게 설명하느냐는 사람마다 달라서 중복이 곧 다양성이 됩니다. 등록 안 된 수가 들어오면 갈래로 자동 그룹화됩니다.

**설계 원칙 두 가지.**

* **승인은 "존재"가 아니라 "승격"에 겁니다.** 설명은 쓰는 즉시 게시판에 존재하고, 기본 노출(수 아래 상위 3~5개)로 올라갈 때만 관리자 승인을 거칩니다. 그래야 검토 시간이 사용자 수와 무관하게 일정합니다.
* **좋아요는 아무것도 발행하지 않습니다.** 검토 큐의 정렬 기준일 뿐입니다. 그래서 앱에서는 로그인 없이 기기 단위로 좋아요를 받아도 손해가 "목록 순서가 흔들림"에 그칩니다.

수 아래 표시는 **관리자가 쓴 공식 설명이 맨 위**, 그 아래 커뮤니티 베스트입니다.

| 구분 | 앱 (모바일) | 사이트 (PC) |
| --- | --- | --- |
| 승격된 설명 읽기 | ✅ 오프라인 캐시 | ✅ |
| 게시판 — 전체 설명 | ✅ 온디맨드 | ✅ |
| 좋아요 · 신고 | ✅ 로그인 없이 | ✅ |
| 계정 · 등록 · 검수 | ❌ (링크로 유도) | ✅ |

등록·검수를 웹으로 뺀 이유는 긴 글을 PC로 쓰는 게 편하고, Expo에 소셜 로그인을 붙이는 비용이 웹보다 훨씬 크며, 자주 바뀔 기능을 스토어 심사 없이 배포할 수 있기 때문입니다. 다만 **앱이 UGC를 표시하는 이상 앱스토어 지침 1.2는 그대로 적용**되므로 신고·차단 수단과 처리 정책은 앱에도 있어야 합니다.

### 지금 동작하는 것 (로컬 검수)

계정도 서버도 없는 상태에서 **화면과 규칙을 먼저** 세워 둔 단계입니다. DB로 옮겨도 규칙은 그대로입니다.

* **앱** — 수 설명 아래 `💬 유저 설명` 칸을 누르면 펼쳐져, **승격된** 설명이 좋아요순 상위 3개까지 보입니다. 같은 칸에서 이 수에 설명을 제안하고(기기에 보관), 공유 시트로 내보내고, **라인 등록기** 링크로 넘어갑니다. 표시 데이터는 `mobile/src/data/openingCommunityNotes.json`이고, 라인이 아니라 **포지션**에 붙으므로 같은 수를 지나는 다른 라인에서도 같은 설명이 보입니다.
* **검수** — 편집기(`npm run author`)에 붙여넣기·승인·좋아요 수·삭제가 붙었습니다. 승인 전에는 파일에만 있고 앱에는 없습니다.
* **아직 아닌 것** — 원격 제출·좋아요·신고. 제안은 공유 시트로 사람이 옮기고, 좋아요는 검수자가 적는 정렬 값입니다. 앱이 UGC를 표시하는 이상 **신고·차단 수단은 출시 전에 필요합니다**(앱스토어 지침 1.2).
* 라인 등록기 주소는 `EXPO_PUBLIC_AUTHOR_URL`로 바꿉니다(기본 `http://localhost:4599`). 기여 사이트가 생기면 이 값만 그쪽으로 돌리면 됩니다.

**약관이 선행 조건입니다.** 유저 설명을 AI로 요약하면 그 결과물은 원문의 2차적저작물이라, 약관에 요약·편집 라이선스가 없으면 만들 수 없습니다. 저작권이 붙는 건 **설명 문장뿐**이고 수순·라인 이름·엔진 평가값은 보호 대상이 아니므로, 위험은 "남의 글을 붙여넣는 경우"로 좁습니다 — 신고·삭제 이력과 작성자 신원 확보가 실질적인 방어선입니다.
