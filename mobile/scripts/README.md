# 오프닝 데이터 도구

앱이 읽는 오프닝 데이터를 만드는 **개발용 도구 두 개**입니다. 앱 번들에는 들어가지 않고, 결과물(JSON·생성 TS)만 커밋됩니다.

| 명령 | 하는 일 |
| --- | --- |
| `npm run author` | 로컬 웹 편집기 — 설명·레퍼런스 작성, 오프닝/갈래 추가·수정·삭제 |
| `npm run gen:openings` | 스톡피쉬로 수별 등급·평가값을 미리 구움 |

두 도구가 `mobile/` 안에 있는 이유는 편집 대상이 전부 `mobile/src/data/`이고 `chess.js`도 앱 것을 그대로 쓰기 때문입니다. 별도 패키지로 빼면 의존성과 상대경로만 늘고 얻는 게 없습니다. **여기는 큐레이션 데이터 전용**이고, 사용자 제출 콘텐츠의 검수는 온라인이어야 하므로 나중에 별도 웹으로 갈라집니다([루트 README](../../README.md#커뮤니티-라인-계획)).

## 데이터 파일

편집기가 쓰는 파일과, 앱이 그걸 합치는 지점입니다.

| 파일 | 내용 | 편집 방법 |
| --- | --- | --- |
| `openings.ts` | 큐레이션 오프닝·라인 원본 | **손으로** (도구가 고치지 않음) |
| `openingNotes.json` | 수별 설명 + 레퍼런스 | 편집기 |
| `openingsAuthored.json` | 편집기로 추가한 오프닝 | 편집기 |
| `openingBranches.json` | 편집기로 추가한 갈래 | 편집기 |
| `openingEdits.json` | 이름 변경·삭제 오버라이드 | 편집기 |
| `openingAnnotations.generated.ts` | 엔진 등급·평가값 | `gen:openings` |

`openingsRuntime.ts`가 위 넷(`openings.ts` + authored + branches + edits)을 합친 결과가 앱이 보는 오프닝 목록입니다.

## 편집기 (`npm run author`)

```bash
cd mobile && npm run author      # http://localhost:4599
```

Docker·Expo와 무관한 별도 서버(포트 4599)입니다. 브라우저가 파일을 못 쓰니 그걸 대신하는 게 존재 이유입니다.

오프닝/라인을 고르면 보드가 수순을 따라갑니다. 라인 목록은 앱과 똑같이 **트리로 들여쓰기**돼 있어(투 나이츠 디펜스 › 나이트 어택 › 폴레리오 디펜스) 어디서 갈라져 나왔는지 그대로 보입니다. 수마다 설명과 레퍼런스(표시명·URL·종류)를 편집해 **저장**하면 `openingNotes.json`에 바로 기록됩니다(파일 순서는 오프닝→라인→ply로 자동 정렬). "Lichess 분석에서 열기"로 현재 포지션을 익스플로러에서 확인할 수 있습니다.

**수의 등급이 보드에 표시됩니다.** 방금 둔 수가 도착한 칸에 앱과 같은 색·기호의 배지가 붙고(`!!` `!` `?!` `?` `??`), 출발 칸은 옅게 표시되며, 아래 수순 스트립도 등급 색으로 물듭니다. 정석 수(`good`)에는 아무것도 붙지 않습니다 — 평범한 수마다 배지를 달면 아무 말도 하지 않는 셈이라서요. 등급은 `openingAnnotations.generated.ts`에서 읽고, 이 파일 역시 바뀔 때마다 다시 읽으므로 `npm run gen:openings`를 돌린 뒤 새로고침만 하면 반영됩니다. 생성 파일에 없는 라인(직접 추가한 갈래)은 SAN 접미사만으로 등급을 매깁니다 — 앱의 `annotationsFor`와 같은 규칙입니다.

`openings.ts`는 **바뀔 때마다 다시 읽습니다.** 라인을 손으로 추가했으면 브라우저만 새로고침하면 되고, 서버를 재시작할 필요가 없습니다.

### 새 오프닝 추가

**"＋ 새 오프닝"** — 시작 포지션부터 보드로 메인라인을 두고 이름·ECO·카테고리·공부하는 색·설명을 붙여 저장합니다. `openingsAuthored.json`에 기록되고 앱 목록에 바로 나타납니다. 변형은 저장 후 갈래로 붙이고, 수 평가는 `gen:openings`를 돌려야 채워집니다(그전까지는 배지 없이 해설만).

### 새 갈래(상대 응수) 추가·수정

**"＋ 새 갈래 추가"** — 지금 위치부터 보드에서 직접 수를 두어 새 라인을 만들고 이름·종류를 붙여 저장합니다. `openingBranches.json`에 저장되고 앱 트리에 즉시 나타나며, 사용자가 그 수를 두면 큐레이션 변형과 똑같이 라인 이름이 뜨고 학습됩니다.

추가한 갈래를 고른 상태에서 **"✎ 이 갈래 수정"** 을 누르면 수순이 보드에 다시 올라옵니다. 수를 더 두거나 되돌리고 이름·종류·등급을 고쳐 덮어쓸 수 있습니다 — **id를 유지하므로 달아둔 설명은 그대로 남습니다.**

### 수 등급

수순의 각 수를 누르면 등급이 순환합니다.

| | | |
| --- | --- | --- |
| `!` 좋은 수 | `!!` 탁월수 | `?!` 부정확 |
| `?` 실수 | `??` 블런더 | (없음) 평범 |

`openings.ts`와 같은 방식으로 SAN 뒤에 접미사로 저장되고(`Nxd5?`), 앱이 `authoredQuality`로 읽어 배지로 띄웁니다. 갈래는 `gen:openings` 대상이 아니라 여기서 붙인 등급이 그대로 유지됩니다. 모든 수는 저장 시 chess.js로 합법성을 검증하며, 등급 접미사는 검증 전에 떼고 저장할 때 붙인 채로 둡니다.

### 이름 변경과 삭제

오른쪽 아래에서 지금 보고 있는 라인·오프닝의 이름을 바꾸거나 삭제합니다. 저장 위치가 대상에 따라 다릅니다.

| 대상 | 이름 변경 | 삭제 |
| --- | --- | --- |
| 편집기로 추가한 오프닝·갈래 | `openingEdits.json` | 원본 파일에서 **완전히 제거** (노트도 함께) |
| `openings.ts`의 큐레이션 데이터 | `openingEdits.json` | `openingEdits.json`에 `hidden`으로 기록 |

큐레이션 데이터를 도구가 직접 고치지 않는 이유는 `openings.ts`가 **손으로 쓰는 원본**이기 때문입니다. 정규식으로 TS 소스를 고쳐 쓰는 건 위험하니, 옆에 둔 오버라이드 파일을 런타임에 덧씌웁니다. **되돌리려면 `openingEdits.json`에서 해당 항목을 지우세요** — 숨긴 항목은 편집기에도 안 보이므로 복구는 파일을 직접 고치는 쪽입니다.

라인의 노트는 라인을 지워도 `openingNotes.json`에 남습니다. 파일 정리(canonicalize)는 **순서만 바꾸고 아무것도 지우지 않습니다** — 라인 id 하나 바꿨다고 손으로 쓴 글이 조용히 사라지면 복구할 방법이 없기 때문입니다. 고아 노트는 파일 끝에 모입니다.

마지막 남은 라인 하나는 지울 수 없습니다. 오프닝에는 메인라인이 반드시 있어야 하므로, 그때는 오프닝을 지우세요.

## 수 평가 굽기 (`npm run gen:openings`)

수 하나하나를 앱에서 판단하면 라인 하나에 엔진 호출이 수십 번 필요합니다. 그래서 평가는 **개발 PC에서 미리 굽고 결과를 커밋합니다.** 앱은 그 파일을 읽기만 하므로 오프닝 학습 중 네트워크·엔진 호출이 **0회**입니다.

```bash
npm run gen:openings                       # STOCKFISH_PATH 기본값 /usr/games/stockfish
STOCKFISH_PATH=... DEPTH=20 npm run gen:openings
```

`generate-opening-annotations.mts`가 모든 라인의 모든 포지션을 depth 18로 평가해 `openingAnnotations.generated.ts`를 덮어씁니다. `openings.ts`의 수순을 고쳤다면 다시 돌리고 **생성 파일까지 함께 커밋**하세요.

엔진을 켜기 **전에** 전 라인을 chess.js로 재생하므로, 불법 수가 있으면 즉시 실패합니다. 별도 검증 절차가 아니라 이 명령이 곧 검증입니다.

등급은 두 층에서 나옵니다.

* **엔진이 매기는 것** — 직전 포지션 대비 손해 본 센티폰(90 이하 정석 / 160 이하 `?!` / 300 이하 `?` / 그 이상 `??`). 대국 복기보다 기준이 관대한데, 정석 수 사이의 60cp 차이는 취향이거나 탐색 노이즈라 학습자에게 `?!`를 붙일 이유가 못 되기 때문입니다. 이미 승부가 갈린 국면(±400 이상)의 등락은 반영하지 않습니다 — 트랩의 결말마다 빨간 배지가 붙어버립니다.
* **사람이 매기는 것** — `!!`와 `!`는 엔진이 절대 만들지 않습니다. 엔진은 "얼마를 잃었나"만 알 뿐 "찾기 어려웠나"는 모릅니다. SAN 접미사가 엔진 판정을 덮어씁니다.

## 저작권

설명 텍스트는 **직접 쓰거나, CC0 데이터(Lichess 게임 DB·익스플로러)·확립된 이론에 근거해** 작성하세요. 남의 Lichess 스터디나 블로그 **문장을 복사하면 저작권 침해**입니다 — 저작권은 작성자에게 있고 제3자에게 넘어가지 않습니다. 참고한 곳은 `refs`에 **링크로만** 답니다.

체스에서 저작권이 붙는 건 **문장뿐**입니다. 수순·라인·오프닝 이름은 사실이고, 엔진 평가값은 기계 산출물이라 어느 쪽도 보호 대상이 아닙니다.
